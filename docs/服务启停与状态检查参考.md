# 服务启停与状态检查参考

## 1. 目标

这份文档整理本项目在 Windows 本机环境下实现以下能力时的最终方案与踩坑记录：

- 启动
- 停止
- 重启
- 状态检查

适用场景：

- 项目目录可能包含中文路径
- 使用本地 Python / `.venv`
- 服务只开放给本机 `127.0.0.1`
- 希望提供“用户双击即可操作”的 `.bat` 入口

这份方案的核心原则是：

- `.bat` 尽量保持最薄
- 真正逻辑放到 `.ps1`
- 健康检查优先用 `curl.exe`
- 启动成功不能只看“进程拉起”，必须验证 `healthz`
- 重启不单独维护一套逻辑，而是复用“停止 + 启动”

---

## 2. 最终推荐结构

推荐将服务控制拆成两层：

### 2.1 用户入口层

用于双击执行，保持尽可能简单：

- `启动监控服务.bat`
- `停止监控服务.bat`
- `重启监控服务.bat`

职责：

- 切换到项目目录
- 调用固定路径的 PowerShell
- 执行对应的 `.ps1`
- 显示最基本的成功/失败提示

### 2.2 PowerShell 逻辑层

用于承载真实逻辑：

- `scripts/start_monitor_console.ps1`
- `scripts/stop_monitor_service.ps1`
- `scripts/restart_monitor_service.ps1`

职责：

- 提权
- 读 `.env`
- 解析端口和监听地址
- 选择 Python
- 做健康检查
- 停止旧进程
- 启动新服务
- 输出可读日志

---

## 3. 最终开发方案

## 3.1 启动方案

### 推荐实现

启动采用：

- `.bat` 直接调用 `start_monitor_console.ps1`
- `start_monitor_console.ps1` 内部先判断是否管理员
- 若不是管理员，则自提权重新执行自身
- 提权成功后，由该脚本再拉起一个新的可见 PowerShell 窗口
- 新窗口中前台执行：
  - `python -m uvicorn monitor_app.api:app --host 127.0.0.1 --port 8010`

### 为什么这样做

因为这台机器上最稳定的链路不是“隐藏后台 + pythonw + 多层嵌套启动”，而是：

- 一个明确提权过的 PowerShell
- 再由它拉起一个服务控制台
- 服务在控制台前台运行

好处：

- 能看到服务启动错误
- 不容易被二次 `RunAs`、隐藏进程或父进程退出影响
- 用户能明确知道“窗口开着 = 服务在运行”

### 启动成功判定

不能只看 `Start-Process` 成功，必须做健康检查：

- 轮询 `http://127.0.0.1:8010/healthz`
- 使用 `curl.exe`
- 连续两次通过可视为稳定

---

## 3.2 停止方案

### 推荐实现

停止脚本不要依赖“保存 PID”这一条单独链路，推荐按端口找进程：

1. 从 `.env` 读取 `MONITOR_API_PORT`
2. 先用 `Get-NetTCPConnection -LocalPort <port> -State Listen`
3. 如果拿不到结果，再回退到：
   - `netstat -ano`
4. 找到监听端口对应的 PID
5. 调用 `Stop-Process -Force`
6. 再用：
   - `curl.exe http://127.0.0.1:<port>/healthz`
   - 配合端口监听检查
   确认服务已经不可达

### 为什么不能只用 `Get-NetTCPConnection`

实测在管理员与非管理员混用、或者父子进程层级复杂时，`Get-NetTCPConnection` 有概率拿不到想要的目标。  
因此稳妥方式是：

- `Get-NetTCPConnection` 作为第一选择
- `netstat -ano` 作为兜底

---

## 3.3 重启方案

### 推荐实现

重启不要再单独维护一套“隐藏启动 + 健康检查 + 端口释放”的完整逻辑。  
最稳的做法是：

- `restart_monitor_service.ps1` 只做两步：
  1. 调用 `stop_monitor_service.ps1`
  2. 调用 `start_monitor_console.ps1`

### 这样做的好处

- 启动逻辑只有一份
- 停止逻辑只有一份
- 后续修启动/停止时，重启天然同步
- 减少“启动成功但重启失败”这类分叉问题

---

## 3.4 状态检查方案

本项目当前重点先把“启动 / 停止 / 重启”做稳。  
状态检查建议单独做一套轻量脚本，不建议直接依赖服务页面是否打开。

### 推荐状态检查输出

状态检查应至少输出以下 4 类结果：

1. `running-healthy`
   - 端口存在监听
   - `healthz` 返回 200

2. `running-unhealthy`
   - 端口存在监听
   - 但 `healthz` 超时或非 200

3. `not-running`
   - 没有监听端口

4. `unknown`
   - 脚本执行异常或无法判断

### 推荐检查步骤

1. 从 `.env` 读取 `MONITOR_API_HOST` / `MONITOR_API_PORT`
2. 检查端口是否有监听
3. 若有监听，再调用：
   - `curl.exe -fsS --max-time 3 http://127.0.0.1:<port>/healthz`
4. 最后组合输出：
   - 运行状态
   - PID
   - URL
   - 最近一次检查结果

### 为什么状态检查必须同时看“端口 + healthz”

因为我们实际遇到过：

- 端口在监听
- 但服务实际上不可用
- 或者脚本误判服务已经启动

所以：

- 只看进程名不够
- 只看监听端口不够
- 必须同时验证 HTTP 健康响应

## 4. 这次踩过的主要坑

## 4.1 `.bat` 不能太重

### 问题

如果 `.bat` 中塞入过长的 PowerShell 命令、复杂引号、中文路径拼接，Windows 下非常容易出现：

- 参数拆裂
- UAC 链路不稳定
- 路径解析错误
- 看起来执行了，实际没真正拉起服务

### 结论

`.bat` 只做：

- `cd /d "%~dp0"`
- 调固定 PowerShell
- 调固定 `.ps1`

其它逻辑全部放 `.ps1`

---

## 4.2 `.venv\Scripts\python.exe` 在中文路径下不一定可靠

### 问题

本项目中曾经出现：

- `.venv\Scripts\python.exe -V` 报 `Unable to create process`
- 系统 Python 路径在 `AppData` 下时，直接调用出现 `Access is denied`

### 本质

虚拟环境启动器依赖 `pyvenv.cfg` 中记录的宿主 Python 路径。  
一旦宿主 Python 路径有权限问题、中文路径兼容问题、或系统策略干预，`.venv\Scripts\python.exe` 可能直接不稳定。

### 结论

要有显式 Python 解析策略：

优先级建议：

1. `.env` 中的 `MONITOR_PYTHON`
2. `.venv` 解析到的宿主 Python
3. `py -3`
4. `python`

并在脚本中做一次 `-V` 探测，不可运行就尽早报错。

---

## 4.3 `Invoke-WebRequest` 在这台机器上不适合做健康检查

### 问题

这次最容易误导人的点就是：

- 服务已经能正常响应
- 但 `Invoke-WebRequest` 会报超时
- 造成脚本误判为“服务没起来”

### 实际表现

同一时刻：

- `curl.exe http://127.0.0.1:8010/healthz` 返回 200
- `Invoke-WebRequest http://127.0.0.1:8010/healthz` 却可能报超时

### 结论

Windows 本机服务的健康检查优先使用：

- `curl.exe -fsS --max-time 3`

不要再把 `Invoke-WebRequest` 当作唯一健康检查手段。

---

## 4.4 隐藏后台链路不一定比可见控制台更稳

### 一开始的直觉

很多项目会默认觉得：

- `pythonw.exe`
- 隐藏 PowerShell
- 后台进程

看起来更“服务化”。

### 实际问题

在这台机器上，隐藏链路叠加：

- 提权
- 中文路径
- `AppData` 下 Python
- 父子 PowerShell 进程

后，调试和稳定性都明显变差。

### 结论

对于本机自用、loopback-only 的监控服务，优先选择：

- 提权后的可见服务控制台

它的可观测性和稳定性反而更高。

---

## 4.5 不能只看“进程创建成功”

### 问题

`Start-Process` 成功并不代表服务真的可用。

可能出现：

- 进程刚起来就退出
- 端口未监听
- 端口已监听但服务未响应

### 结论

启动完成条件必须是：

- 健康检查通过
- 最好再二次确认一次，确保稳定

---

## 4.6 停止旧服务时要考虑“权限层级”

### 问题

如果服务是管理员权限启动的，那么普通 PowerShell 下去 `Stop-Process`，会出现：

- `Access is denied`

### 结论

停止脚本本身也必须支持自提权，不能假设调用方已经有管理员权限。

---

## 5. 推荐的可复用脚本职责划分

## 5.1 启动

### `启动服务.bat`

只做入口：

- 调 `start_service.ps1`

### `start_service.ps1`

负责：

- 自提权
- 读环境变量
- 解析 Python
- 清理旧监听
- 打开服务控制台
- 用 `curl.exe` 轮询 `healthz`

---

## 5.2 停止

### `停止服务.bat`

只做入口：

- 调 `stop_service.ps1`

### `stop_service.ps1`

负责：

- 自提权
- 按端口找 PID
- 结束进程
- 用 `curl.exe` + 端口检测确认服务已停止

---

## 5.3 重启

### `重启服务.bat`

只做入口：

- 调 `restart_service.ps1`

### `restart_service.ps1`

负责：

- 自提权
- 先执行停止
- 再执行启动

---

## 5.4 状态检查

### `状态检查服务.bat`

只做入口：

- 调 `status_service.ps1`

### `status_service.ps1`

建议负责：

- 自提权
- 判断端口监听
- 调 `curl.exe` 检查 `healthz`
- 输出：
  - healthy / unhealthy / not-running
  - PID
  - URL

---

## 6. 给其它项目的直接结论

如果你要把这套经验迁到其它 Windows 本地项目，推荐直接照下面做：

1. `.bat` 保持最薄
2. 所有复杂逻辑放 `.ps1`
3. 启动、停止、重启三者复用同一套底层逻辑
4. 健康检查统一用 `curl.exe`
5. 状态检查同时看“端口监听 + /healthz”
6. 对中文路径、`AppData` Python、`.venv` 启动器异常做好兜底
7. 对“管理员权限启动的服务”默认要求停止脚本也能提权
8. 对本机服务优先采用“可见控制台”而不是一开始就追求隐藏后台
