# 访问控制模块接入说明

这份说明对应当前项目已经落地的访问控制模块，用于后续在本项目或仓内其他项目复用。

## 目标

访问控制模块统一处理：

- 白名单 IP 免登录
- 游客 / 管理员双密码登录
- 会话 cookie 与 CSRF
- break-glass 本机应急入口
- 服务端 capability 权限控制
- 登录页与业务 API 的差异化拦截

## 配置文件

默认配置路径：

- `config/access_control.json`

当前推荐结构：

```json
{
  "enabled": true,
  "whitelist_ips": ["127.0.0.1", "::1"],
  "allow_plaintext_secrets": false,
  "cookie_secure_mode": "auto",
  "guest_password_secret_ref": "access_control.guest_password",
  "admin_password_secret_ref": "access_control.admin_password",
  "session_secret_secret_ref": "access_control.session_secret"
}
```

说明：

- `whitelist_ips` 只接受合法 IPv4 / IPv6
- `cookie_secure_mode` 只允许：
  - `auto`
  - `always`
  - `never`
- 默认采用 `refs-only` 模式，不再推荐明文密码

## Secret 依赖

访问控制模块本身不负责明文 secret 的长期存储，而是依赖项目统一的加密 secret 仓库。

常用 ref：

- `access_control.guest_password`
- `access_control.admin_password`
- `access_control.session_secret`

主密钥建议通过：

- `MONITOR_MASTER_KEY_FILE`

提供，而不是直接写入业务配置。

## 当前行为

### 白名单

- 命中白名单 IP：直接放行
- 默认 loopback 也视为白名单管理员来源：
  - `127.0.0.1`
  - `::1`

### 登录模式

- `enabled=false`：
  - 服务开放
  - 不要求登录
- `enabled=true`：
  - 非白名单请求必须登录

### 角色

- `guest`
  - `monitor.read`
  - `funding.read`
- `admin`
  - 全部 capability

## 关键接口

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/session`
- `GET /api/auth/audit`
- `GET /api/auth/break-glass/challenge`
- `POST /api/auth/break-glass/disable`

### break-glass

当前 break-glass 已经改成 challenge + confirm：

1. 先调用：
   - `GET /api/auth/break-glass/challenge`
2. 拿到一次性 `nonce`
3. 再调用：
   - `POST /api/auth/break-glass/disable`
   - 请求体必须带 `nonce`

限制：

- 只允许 loopback 来源使用
- break-glass 生效后也只对 loopback 提供临时管理员上下文
- 远程请求不会被一起放开

## 路由策略

模块当前采用 deny-by-default。

对以下命名空间：

- `/api/*`
- `/stream/*`
- `/static/*`
- `/public/*`

如果运行时没有声明策略，会直接拒绝，而不是默认放行。

启动时还会执行：

- `route_capability_snapshot(app)`

如果存在未声明 capability 的业务路由，服务启动直接失败。

如果本地 secret 仓库已经就绪，还会在启动阶段顺带检查：

- 配置中的 secret ref 是否都能在仓库里解出来
- 是否存在“配置已引用，但仓库缺值”的情况

## 登录页与公共资源

登录页入口：

- `GET /login`

登录页公共资源前缀：

- `/public/login/*`

这段前缀是匿名可访问的，供后续放 logo、CSS、JS、favicon 等资源。  
业务资源仍然走：

- `/static/*`

并继续受权限保护。

## 会话与 CSRF

登录成功后，服务端会下发会话 cookie。  
所有非 GET 写请求都必须附带：

- `X-CSRF-Token`

当前 `session`、`whitelist`、`break_glass` 三类管理员上下文，都会要求通过 CSRF 校验后才能执行写操作。

另外还会做一层浏览器同源校验：

- 优先检查 `Origin`
- 缺失时回退检查 `Referer`
- 非同源写请求会直接拒绝

当前默认空闲超时：

- `admin`：30 分钟
- `guest`：120 分钟

超时后会话会失效，并要求重新登录。

登录失败限流状态会持久化到本地 SQLite，服务重启后仍然保留当前冷却窗口。

## 前端接线要求

页面启动时先请求：

- `/api/auth/session`

根据返回值控制：

- 当前角色显示
- 游客按钮禁用
- 写操作请求附带 CSRF token

未登录时：

- 页面请求重定向到 `/login`
- API 请求返回 JSON 认证错误

## 仓内复用建议

如果其他项目想复用这套访问控制，最小接入步骤是：

1. 启动时创建 `AccessControlService`
2. 在中间件里执行：
   - `build_context(request)`
   - `authorize_request(request, context)`
3. 复用认证接口：
   - `login(...)`
   - `logout(...)`
   - `session_payload_for_response(...)`
   - `issue_break_glass_challenge(...)`
   - `break_glass_disable(...)`
4. 为所有业务路由补 capability 声明
5. 保证 `route_capability_snapshot()` 启动自检通过

## 注意事项

- 当前只认直连 client IP，不信任代理头
- 配置改动后需要重启服务生效
- 生产环境建议坚持 `allow_plaintext_secrets=false`
- 未来挂到 HTTPS 或反向代理时，建议把 `cookie_secure_mode` 调整到更严格策略
